# Component — Scenario player

Consumes scenario definitions and fires the corresponding events into the [dealFeed](deal-feed.md) at the right times. Used internally by the dealFeed — one player instance per dealFeed singleton.

File: `src/services/scenarios/player.ts`. Definitions: `src/services/scenarios/definitions.ts`. Tests: `player.test.ts`, `definitions.test.ts`.

## API

```typescript
function createScenarioPlayer(deps: {
  emit: (event: DealEvent) => void;
  now?: () => number;
  generateDealId?: () => string;
  acceptOrReject?: (dealId: string) => boolean;  // FXSW-090: pin the coin-flip
}): ScenarioPlayer;

interface ScenarioPlayer {
  inject(scenarioId: ScenarioId, overrides?: ScenarioOverrides): void; // overrides: { tenor, instrumentType, farTenor }
  reset(): void;
  forgetDeal(dealId: string): void;       // FXSW-090: drop a deal's pending timers + gates
  notifyDealState(dealId: string, siState: string): void;
}
```

Factory-with-injectable-seams pattern: `emit`, `now`, `generateDealId`, and `acceptOrReject` can all be pinned for tests so dealIds, timestamps, and the accept/reject outcome are deterministic. The dealFeed instantiates one player at module load with `emit` wired to the fan-out.

## `buildDeal` — instrument-aware injection (v4)

`inject(scenarioId, overrides)` constructs the deal via `buildDeal`, which sets `instrumentType` (from the override, the scenario default, or `defaultInstrumentForTenor(tenor)`) and handles instrument-specific tenor shaping in one place:

- **NDF** — must carry a forward tenor; a SPOT request is coerced to the shortest forward (`FORWARD_TENORS[0]`). It must also be struck on a **non-deliverable pair**: if the scenario's pair is deliverable (EURUSD/GBPUSD/USDJPY), `buildDeal` coerces it to an NDF pair (`NDF_PAIRS[0]`, currently `USDINR`) rather than quoting an NDF on a deliverable currency. `isNdfPair` / `NDF_PAIRS` live in `src/services/feed/types.ts`. See [features/ndf.md](../features/ndf.md).
- **SWAP** — calls `resolveSwapLegs(near, far?)` (in `src/types/deal.ts`) to build `[NEAR, FAR]` with far strictly later, coercing a missing/out-of-order far to the shortest valid far (last-tenor near steps back). `Deal.tenor` mirrors the NEAR leg; when the legs were coerced, the original request is recorded on `Deal.swapRequested` for the "legs adjusted" note. See [features/swaps.md](../features/swaps.md).

Single source of coercion: the player owns it, so every entry point gets identical behaviour.

## Follow-up triggers

Each scenario definition is a `NEW_*_DEAL` event plus zero or more `ScenarioFollowUp` entries. Two kinds of follow-up:

- **Time-gated**: `{ kind: 'delay', ms: 2000, event: { type: 'CLIENT_ACCEPT' } }` — fires `ms` after `inject()` via `setTimeout`. Used by [happy-path-esp.md](../scenarios/happy-path-esp.md).
- **State-gated**: `{ kind: 'after-si-state', state: 'Quoted', delayMs: 1500, event: { type: 'CLIENT_ACCEPT' } }` — registers a `Gate` that arms when `notifyDealState(dealId, 'Quoted')` lands; then schedules the event `delayMs` after arming. Used by [off-hours-intervention.md](../scenarios/off-hours-intervention.md), [size-limit-margin-tune.md](../scenarios/size-limit-margin-tune.md).

The dealId in each follow-up's event refers to whichever deal was generated by the leading `NEW_*_DEAL` for that `inject()` call — the player threads it through automatically.

### Randomized outcome event — `CLIENT_ACCEPT_OR_REJECT` (Phase 6.1)

`FollowUpEvent` (in `src/types/scenario.ts`) gained a `CLIENT_ACCEPT_OR_REJECT` member. It is **not** a new trigger kind — it rides the existing `delay` / `after-si-state` triggers — but a different *outcome* that is resolved **at scheduling time**, inside `player.ts:buildFollowUpEvent`:

So a single declarative follow-up entry fans out to either a `CLIENT_ACCEPT` or a `CLIENT_REJECT` deal event on a coin-flip. Used by [`CREDIT_BREACH`](../scenarios/credit-breach.md) to give the (non-recommended) trader-quote path a terminal outcome.

**Seeded + injectable (FXSW-090).** The coin-flip is **no longer `Math.random()`** — it is resolved by `acceptOrReject(dealId)`, which defaults to a **seeded PRNG keyed off the deal id** (`makeRng(hashSeed(dealId) ^ PLAYER_SEED)`, reusing `services/feed/rng`). A given deal's outcome is therefore reproducible, and tests can pin it via the injectable `acceptOrReject` dep (and `generateDealId`). See [ADR-0016](../decisions/ADR-0016-ga-core-determinism.md). *(This supersedes the earlier "no injectable random seam / Math.random direct" behaviour.)*

## Executed side for two-way requests (FXSW-092)

A two-way request (`side: 'BOTH'`) is quoted on both sides but the client deals on exactly **one**. The player resolves which, so the `CLIENT_ACCEPT` event can carry an `executedSide` (`BID|ASK`, the `DealtSide` type in `src/lib/quoteSide.ts`):

- **At inject**, the player records each deal's quotable side in a `dealSides` map: `dealSides.set(dealId, quoteSideFor(deal.side, deal.dealtCcy))`.
- **`resolveExecutedSide(dealId)`** then decides the dealt side when the accept fires:
  - a **one-sided** request (`dealSides` is `BID` or `ASK`) deals on its only quotable side;
  - a **two-way** request's client **picks one via a per-deal seeded flip** — `makeRng(hashSeed(dealId) ^ EXEC_SIDE_SEED)() < 0.5 ? 'BID' : 'ASK'`. `EXEC_SIDE_SEED` is a **separate seed** from the accept/reject coin-flip (`PLAYER_SEED`), so a given deal's executed side is reproducible across runs and pinnable in tests — same determinism pattern as the [accept/reject flip](#randomized-outcome-event--client_accept_or_reject-phase-61) (FXSW-090).
- `buildFollowUpEvent` threads `resolveExecutedSide` into the `CLIENT_ACCEPT` and `CLIENT_ACCEPT_OR_REJECT` events, so the dealt side rides the existing follow-up triggers (no new trigger kind).
- The `dealSides` entry is **cleared in `forgetDeal`** (on archival / removeDeal) and wiped wholesale in `reset()`, mirroring the timer/gate cleanup below.

Downstream, `dealsBootstrap` records the side via `recordExecutedSide` just before confirming, and the [deals store](deals-store.md) snapshots it onto the archived entry (only when Executed) so the [historical detail](../features/historical-detail.md) banner can surface it. See [ADR-0017](../decisions/ADR-0017-swap-markup-model-and-executed-side.md).

## `forgetDeal` + reset cleanup (FXSW-090)

`forgetDeal(dealId)` clears every pending `setTimeout` handle, every unarmed state-gate, **and** the deal's `dealSides` entry, so an archived deal can't fire a stale follow-up and neither the internal `gates` Set nor the `dealSides` map can grow unbounded. `reset()` clears all timers, gates, and recorded sides wholesale (the session wipe). Both keep the simulation leak-free across long demo sessions.

## Reset semantics

`reset()` clears:

- Pending `setTimeout`s for time-gated follow-ups.
- Unarmed `Gate`s for state-gated follow-ups.

After reset, no further callbacks fire — even if `notifyDealState` is called later for a stale dealId.

## Scenarios

The v1 set (surfaced at `?dev=1`):

| ID | Lead event | Follow-ups |
|---|---|---|
| `HAPPY_PATH_ESP` | `NEW_ESP_DEAL` | `CLIENT_ACCEPT` at t+2000 |
| `OFF_HOURS_INTERVENTION` | `NEW_SI_DEAL` | `CLIENT_ACCEPT` 1500ms after SI reaches `Quoted` |
| `CREDIT_BREACH` | `NEW_SI_DEAL` | `CLIENT_ACCEPT_OR_REJECT` 1500ms after SI reaches `Quoted` (Phase 6.1 — covers the non-recommended trader-quote path; trader Reject is still the recommended action) |
| `SIZE_LIMIT_MARGIN_TUNE` | `NEW_SI_DEAL` | `CLIENT_ACCEPT` 2000ms after SI reaches `Quoted` |
| `RELEASE_PATH` | `NEW_SI_DEAL` | `CLIENT_ACCEPT` 1500ms after SI reaches `Quoted` (Phase 6.1 — only fires on the alternative quote path; the Release action never reaches `Quoted`) |

Phase 6.1's two follow-up additions (`CREDIT_BREACH`, `RELEASE_PATH`) close the previous gap where a trader who quoted instead of taking the recommended action left the deal hanging in `Quoted` forever.

A v2 set (surfaced only at `?dev=v2`) adds two further definitions — `BOTH_SIDED_INQUIRY`, `QUOTE_DEALT_INQUIRY`. The id groupings live in `src/types/scenario.ts` as `V1_SCENARIO_IDS` / `V2_SCENARIO_IDS` / `SCENARIO_IDS`; the [dev injector](../components/dev-injector.md) chooses the visible set from the dev version.

Full payloads (client, account, pair, side, notional, reasons) live in `src/services/scenarios/definitions.ts` and per-scenario wiki pages — see the [Scenarios section of the index](../index.md#scenarios). The `definitions.test.ts` round-trip verifies the definitions match the docs.

## State-gate type loosening

The state-gate's `state` field is typed as a plain `string`, not the closed SI-state union. The full SI state set landed in FXSW-010; tying the gate to the not-yet-complete enum at FXSW-008 would have created churn. The trade-off: no typo protection on the gate name. The `definitions.test.ts` round-trip is the practical guard. See `docs/dev-log.md` FXSW-008 entry.
