# Component ‚Äî Scenario player

Consumes scenario definitions and fires the corresponding events into the [dealFeed](deal-feed.md) at the right times. Used internally by the dealFeed ‚Äî one player instance per dealFeed singleton.

File: `src/services/scenarios/player.ts`. Definitions: `src/services/scenarios/definitions.ts`. Tests: `player.test.ts`, `definitions.test.ts`.

## API

```typescript
function createScenarioPlayer(deps: {
  emit: (event: DealEvent) => void;
  now?: () => number;
  generateDealId?: () => string;
  acceptOrReject?: (dealId: string) => boolean;  // FXSW-090: pin the coin-flip
}): ScenarioPlayer;

interface ScenarioPlayer {
  inject(scenarioId: ScenarioId, overrides?: ScenarioOverrides): void; // overrides: { tenor, instrumentType, farTenor }
  reset(): void;
  forgetDeal(dealId: string): void;       // FXSW-090: drop a deal's pending timers + gates
  notifyDealState(dealId: string, siState: string): void;
}
```

Factory-with-injectable-seams pattern: `emit`, `now`, `generateDealId`, and `acceptOrReject` can all be pinned for tests so dealIds, timestamps, and the accept/reject outcome are deterministic. The dealFeed instantiates one player at module load with `emit` wired to the fan-out.

## `buildDeal` ‚Äî instrument-aware injection (v4)

`inject(scenarioId, overrides)` constructs the deal via `buildDeal`, which sets `instrumentType` (from the override, the scenario default, or `defaultInstrumentForTenor(tenor)`) and handles instrument-specific tenor shaping in one place:

- **NDF** ‚Äî must carry a forward tenor; a SPOT request is coerced to the shortest forward (`FORWARD_TENORS[0]`). See [features/ndf.md](../features/ndf.md).
- **SWAP** ‚Äî calls `resolveSwapLegs(near, far?)` (in `src/types/deal.ts`) to build `[NEAR, FAR]` with far strictly later, coercing a missing/out-of-order far to the shortest valid far (last-tenor near steps back). `Deal.tenor` mirrors the NEAR leg; when the legs were coerced, the original request is recorded on `Deal.swapRequested` for the "legs adjusted" note. See [features/swaps.md](../features/swaps.md).

Single source of coercion: the player owns it, so every entry point gets identical behaviour.

## Follow-up triggers

Each scenario definition is a `NEW_*_DEAL` event plus zero or more `ScenarioFollowUp` entries. Two kinds of follow-up:

- **Time-gated**: `{ kind: 'delay', ms: 2000, event: { type: 'CLIENT_ACCEPT' } }` ‚Äî fires `ms` after `inject()` via `setTimeout`. Used by [happy-path-esp.md](../scenarios/happy-path-esp.md).
- **State-gated**: `{ kind: 'after-si-state', state: 'Quoted', delayMs: 1500, event: { type: 'CLIENT_ACCEPT' } }` ‚Äî registers a `Gate` that arms when `notifyDealState(dealId, 'Quoted')` lands; then schedules the event `delayMs` after arming. Used by [off-hours-intervention.md](../scenarios/off-hours-intervention.md), [size-limit-margin-tune.md](../scenarios/size-limit-margin-tune.md).

The dealId in each follow-up's event refers to whichever deal was generated by the leading `NEW_*_DEAL` for that `inject()` call ‚Äî the player threads it through automatically.

### Randomized outcome event ‚Äî `CLIENT_ACCEPT_OR_REJECT` (Phase 6.1)

`FollowUpEvent` (in `src/types/scenario.ts`) gained a `CLIENT_ACCEPT_OR_REJECT` member. It is **not** a new trigger kind ‚Äî it rides the existing `delay` / `after-si-state` triggers ‚Äî but a different *outcome* that is resolved **at scheduling time**, inside `player.ts:buildFollowUpEvent`:

So a single declarative follow-up entry fans out to either a `CLIENT_ACCEPT` or a `CLIENT_REJECT` deal event on a coin-flip. Used by [`CREDIT_BREACH`](../scenarios/credit-breach.md) to give the (non-recommended) trader-quote path a terminal outcome.

**Seeded + injectable (FXSW-090).** The coin-flip is **no longer `Math.random()`** ‚Äî it is resolved by `acceptOrReject(dealId)`, which defaults to a **seeded PRNG keyed off the deal id** (`makeRng(hashSeed(dealId) ^ PLAYER_SEED)`, reusing `services/feed/rng`). A given deal's outcome is therefore reproducible, and tests can pin it via the injectable `acceptOrReject` dep (and `generateDealId`). See [ADR-0016](../decisions/ADR-0016-ga-core-determinism.md). *(This supersedes the earlier "no injectable random seam / Math.random direct" behaviour.)*

## `forgetDeal` + reset cleanup (FXSW-090)

`forgetDeal(dealId)` clears every pending `setTimeout` handle **and** every unarmed state-gate owned by that deal, so an archived deal can't fire a stale follow-up and the internal `gates` Set can't grow unbounded. `reset()` clears all timers and gates wholesale (the session wipe). Both keep the simulation leak-free across long demo sessions.

## Reset semantics

`reset()` clears:

- Pending `setTimeout`s for time-gated follow-ups.
- Unarmed `Gate`s for state-gated follow-ups.

After reset, no further callbacks fire ‚Äî even if `notifyDealState` is called later for a stale dealId.

## Scenarios

The v1 set (surfaced at `?dev=1`):

| ID | Lead event | Follow-ups |
|---|---|---|
| `HAPPY_PATH_ESP` | `NEW_ESP_DEAL` | `CLIENT_ACCEPT` at t+2000 |
| `OFF_HOURS_INTERVENTION` | `NEW_SI_DEAL` | `CLIENT_ACCEPT` 1500ms after SI reaches `Quoted` |
| `CREDIT_BREACH` | `NEW_SI_DEAL` | `CLIENT_ACCEPT_OR_REJECT` 1500ms after SI reaches `Quoted` (Phase 6.1 ‚Äî covers the non-recommended trader-quote path; trader Reject is still the recommended action) |
| `SIZE_LIMIT_MARGIN_TUNE` | `NEW_SI_DEAL` | `CLIENT_ACCEPT` 2000ms after SI reaches `Quoted` |
| `RELEASE_PATH` | `NEW_SI_DEAL` | `CLIENT_ACCEPT` 1500ms after SI reaches `Quoted` (Phase 6.1 ‚Äî only fires on the alternative quote path; the Release action never reaches `Quoted`) |

Phase 6.1's two follow-up additions (`CREDIT_BREACH`, `RELEASE_PATH`) close the previous gap where a trader who quoted instead of taking the recommended action left the deal hanging in `Quoted` forever.

A v2 set (surfaced only at `?dev=v2`) adds two further definitions ‚Äî `BOTH_SIDED_INQUIRY`, `QUOTE_DEALT_INQUIRY`. The id groupings live in `src/types/scenario.ts` as `V1_SCENARIO_IDS` / `V2_SCENARIO_IDS` / `SCENARIO_IDS`; the [dev injector](../components/dev-injector.md) chooses the visible set from the dev version.

Full payloads (client, account, pair, side, notional, reasons) live in `src/services/scenarios/definitions.ts` and per-scenario wiki pages ‚Äî see the [Scenarios section of the index](../index.md#scenarios). The `definitions.test.ts` round-trip verifies the definitions match the docs.

## State-gate type loosening

The state-gate's `state` field is typed as a plain `string`, not the closed SI-state union. The full SI state set landed in FXSW-010; tying the gate to the not-yet-complete enum at FXSW-008 would have created churn. The trade-off: no typo protection on the gate name. The `definitions.test.ts` round-trip is the practical guard. See `docs/dev-log.md` FXSW-008 entry.
