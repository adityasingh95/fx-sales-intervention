# Component — Scenario player

Consumes scenario definitions and fires the corresponding events into the [dealFeed](deal-feed.md) at the right times. Used internally by the dealFeed — one player instance per dealFeed singleton.

File: `src/services/scenarios/player.ts`. Definitions: `src/services/scenarios/definitions.ts`. Tests: `player.test.ts`, `definitions.test.ts`.

## API

```typescript
function createScenarioPlayer(deps: {
  emit: (event: DealEvent) => void;
  now?: () => number;
  generateDealId?: () => string;
}): ScenarioPlayer;

interface ScenarioPlayer {
  inject(scenarioId: ScenarioId): void;
  reset(): void;
  notifyDealState(dealId: string, siState: string): void;
}
```

Factory-with-injectable-seams pattern: `emit`, `now`, and `generateDealId` can all be pinned for tests so dealIds and timestamps are deterministic. The dealFeed instantiates one player at module load with `emit` wired to the fan-out.

## Follow-up triggers

Each scenario definition is a `NEW_*_DEAL` event plus zero or more `ScenarioFollowUp` entries. Two kinds of follow-up:

- **Time-gated**: `{ kind: 'delay', ms: 2000, event: { type: 'CLIENT_ACCEPT' } }` — fires `ms` after `inject()` via `setTimeout`. Used by [happy-path-esp.md](../scenarios/happy-path-esp.md).
- **State-gated**: `{ kind: 'after-si-state', state: 'Quoted', delayMs: 1500, event: { type: 'CLIENT_ACCEPT' } }` — registers a `Gate` that arms when `notifyDealState(dealId, 'Quoted')` lands; then schedules the event `delayMs` after arming. Used by [off-hours-intervention.md](../scenarios/off-hours-intervention.md), [size-limit-margin-tune.md](../scenarios/size-limit-margin-tune.md).

The dealId in each follow-up's event refers to whichever deal was generated by the leading `NEW_*_DEAL` for that `inject()` call — the player threads it through automatically.

## Reset semantics

`reset()` clears:

- Pending `setTimeout`s for time-gated follow-ups.
- Unarmed `Gate`s for state-gated follow-ups.

After reset, no further callbacks fire — even if `notifyDealState` is called later for a stale dealId.

## Scenarios

Five definitions registered in v1:

| ID | Lead event | Follow-ups |
|---|---|---|
| `HAPPY_PATH_ESP` | `NEW_ESP_DEAL` | `CLIENT_ACCEPT` at t+2000 |
| `OFF_HOURS_INTERVENTION` | `NEW_SI_DEAL` | `CLIENT_ACCEPT` 1500ms after SI reaches `Quoted` |
| `CREDIT_BREACH` | `NEW_SI_DEAL` | (none — trader must reject) |
| `SIZE_LIMIT_MARGIN_TUNE` | `NEW_SI_DEAL` | `CLIENT_ACCEPT` 2000ms after SI reaches `Quoted` |
| `RELEASE_PATH` | `NEW_SI_DEAL` | (none — trader must release) |

Full payloads (client, account, pair, side, notional, reasons) live in `src/services/scenarios/definitions.ts` and per-scenario wiki pages under [scenarios/](../scenarios/). The `definitions.test.ts` round-trip verifies the definitions match the docs.

## State-gate type loosening

The state-gate's `state` field is typed as a plain `string`, not the closed SI-state union. The full SI state set landed in FXSW-010; tying the gate to the not-yet-complete enum at FXSW-008 would have created churn. The trade-off: no typo protection on the gate name. The `definitions.test.ts` round-trip is the practical guard. See `docs/dev-log.md` FXSW-008 entry.
